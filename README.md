# Provenance Guard

A backend system that a creative-sharing platform can plug into to classify whether submitted
text was likely written by a human or generated by AI. For every submission it returns a
structured attribution result, a calibrated confidence score, and a plain-language
**transparency label** to show readers. Creators can **appeal** a classification, and every
decision and appeal is written to a structured **audit log**.

This README is the canonical record of the system's behavior and evidence. The design
rationale and architecture diagrams live in [planning.md](planning.md).

## Setup & running

Requires Python 3.11+ and a [Groq](https://console.groq.com/) API key.

```bash
# 1. install dependencies
python -m venv .venv
.venv/Scripts/python.exe -m pip install -r requirements.txt   # Windows
# source .venv/bin/activate && pip install -r requirements.txt # macOS/Linux

# 2. set your key
echo "GROQ_API_KEY=your_key_here" > .env

# 3. run
.venv/Scripts/python.exe app.py
```

The server starts on `http://127.0.0.1:5000`. This is an **API-only backend** — there is no
web UI. The transparency label is delivered as a text string in the JSON response (the `label`
field); a host platform would render it however it likes.

> **PowerShell note:** `curl` is aliased to `Invoke-WebRequest` and will not accept `-H`/`-d`.
> Use `curl.exe` (the real curl) or `Invoke-RestMethod` instead.

## API endpoints

| Method | Path | Body | Purpose |
|---|---|---|---|
| POST | `/submit` | `{text, creator_id}` | Classify a piece of text; returns attribution, confidence, label, both signals. Rate-limited. |
| POST | `/appeal` | `{content_id, creator_reasoning}` | Contest a classification; sets status to `under_review`. |
| GET | `/appeals` | — | Reviewer queue: all appeals with the original decision snapshot. |
| GET | `/submissions/<content_id>` | — | Full stored record for one submission. |
| GET | `/log` | — | Structured audit log (JSON). |

Example:

```bash
curl.exe -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon as I sat on the porch.", "creator_id": "test-user-1"}'
```

## Architecture

Full diagrams (submission flow + appeal flow, with labeled data on each arrow) are in
[planning.md → Architecture](planning.md#2-architecture). In brief:

```
POST /submit ──raw text──► Signal 1 (Groq LLM judge)  ──┐
                           Signal 2 (stylometric)     ──┤
                                                        ▼
                              combine_scores (0.5/0.5) ──► combined_score (0–1)
                                                        ▼
                              make_label (thresholds)  ──► transparency label
                                                        ▼
                              audit log + submissions store ──► JSON response
```

A submission's text is sent to two independent signals; their scores are averaged into a single
`combined_score`, mapped to one of three label variants by fixed thresholds, written to the
audit log, and returned in the same request. An appeal does **not** re-run detection — it
records the creator's reasoning, flips the submission's status to `under_review`, and appends a
second audit entry referencing the original decision.

Code map: [app.py](app.py) (routes, rate limiting) · [signals.py](signals.py) (both signals +
scoring) · [labels.py](labels.py) (label text) · [store.py](store.py) (submissions) ·
[audit_log.py](audit_log.py) (audit log).

## Detection signals (and why these two)

The pipeline uses **two independent signals** that fail in *different* ways, so that one can
catch the other's mistakes. Single-signal detection was a non-starter precisely because the
strongest single signal (the LLM judge) has a well-known, systematic bias (below) that a second,
mechanically-different signal can partially offset.

### Signal 1 — LLM-as-judge (Groq `llama-3.3-70b-versatile`)

- **What it measures:** holistic authorship cues — idea originality, voice consistency,
  idiosyncrasy, whether phrasing reads as templated/instruction-following. Output:
  `{"ai_likelihood": float 0–1, "rationale": str}`, called at `temperature=0`.
- **Why it works:** general-purpose LLMs have seen enormous amounts of both human and machine
  text and recognize diffuse stylistic patterns (cliché phrasing, over-explanation, hedged
  "balanced" prose) that are hard to write down as explicit rules.
- **Blind spot:** it inherits the judge model's biases. LLM-based detectors are documented to
  over-flag **non-native-English** and simple, formulaic human writing as "AI-like." It is also
  not perfectly deterministic, and its rationale is self-reported (possibly post-hoc).

### Signal 2 — Stylometric heuristics (pure Python, no network)

A weighted sum of three sub-features, each normalized to `[0,1]` in the "AI-like" direction:

| Sub-feature | Measures | Why it differs human vs. AI |
|---|---|---|
| Burstiness | Variance in sentence length | Humans mix short and long sentences; AI output is more uniform. |
| Lexical repetition | `1 − type/token ratio` | AI reuses word choices more; humans vary vocabulary. |
| Discourse-marker density | Rate of stock transitions ("moreover", "in conclusion", …) | Instruction-tuned LLMs overuse these vs. typical creative writing. |

```
signal2 = 0.5 * marker_score + 0.25 * burstiness_score + 0.25 * repetition_score
```

- **Why a heuristic at all:** it is transparent, free, deterministic, needs no network, and —
  critically — fails *independently* of the LLM. It does not understand meaning, so it can't be
  fooled by the same things that fool the LLM.
- **Blind spot:** all three features are length-sensitive and English-specific, are trivially
  gameable ("vary your sentence length"), and conflate deliberate style (poetic refrains, simple
  vocabulary) with machine generation. It captures *surface form only*.
- **Why markers are weighted highest:** see the [confidence-scoring](#confidence-scoring)
  section — on short text the other two features are nearly dead, so an equal mean diluted the
  one feature that actually discriminated.

### If we deployed this for real

- Replace the LLM judge's single self-reported score with **multiple sampled judgments** (or an
  ensemble of different models) and use their spread as an additional uncertainty signal.
- Add a **perplexity / token-probability** signal from a small open model — a mechanically
  different third signal that is much harder to game than surface heuristics.
- Calibrate weights and thresholds against a **labeled validation set** (we have none today, so
  the 0.5/0.5 blend is a deliberate, explainable default, not a tuned result).

## Confidence scoring

`combined_score` is interpreted as **"probability the content is AI-generated"** on a 0–1
scale: `0` = confidently human, `1` = confidently AI, `0.5` = no useful signal. We read both
*direction* (which side of 0.5) and *confidence* (distance from 0.5) off this single number,
which keeps it explainable to a non-technical reader.

```
combined_score = 0.5 * signal1 + 0.5 * signal2
```

| `combined_score` | Label variant |
|---|---|
| `>= 0.70` | likely AI |
| `<= 0.30` | likely human |
| `0.30 < s < 0.70` | uncertain |

The exact score is always returned alongside the label, so the percentage shown to a reader
changes continuously even when the category does not — `0.62` and `0.95` are both AI-leaning but
read very differently.

### How we tested that the scores are meaningful

We ran four deliberately chosen inputs (clearly AI, clearly human, and two borderline cases) and
checked that the combined score varied meaningfully and that all three label bands were
reachable. The scores are **not** a constant:

| Input | Signal 1 (LLM) | Signal 2 (stylo) | **Combined** | Label |
|---|---|---|---|---|
| **Clearly AI** ("…transformative paradigm shift… It is important to note… Furthermore…") | 0.80 | 0.61 | **0.70** | likely AI |
| **Clearly human** ("ok so i finally tried that new ramen place downtown and honestly? underwhelming…") | 0.20 | 0.07 | **0.14** | likely human |
| Borderline — formal human (monetary-policy paragraph) | 0.80 | 0.11 | **0.46** | uncertain |
| Borderline — lightly edited AI (remote-work paragraph) | 0.60 | 0.12 | **0.36** | uncertain |

**Two examples with noticeably different confidence (required):**
- **High-confidence case:** the *clearly human* ramen review scored **0.14** → a confident
  "likely human-written (86% likely human)" label.
- **Lower-confidence case:** the *formal human* monetary-policy paragraph scored **0.46** → an
  honest "we could not confidently determine…" label.

### Why the scoring approach changed during testing (calibration story)

Our first implementation combined Signal 2's three sub-features with a **plain mean**. With that,
the *clearly-AI* input scored only **0.639 → "uncertain"**, and across all four inputs only two
of the three label bands were ever reached. Printing each signal separately revealed why: on
short text (~40–55 words) two of Signal 2's three sub-features are effectively dead — type/token
ratio saturates high (few words repeat in 40), and sentence-length variance is similar across
authors at that length. **Only discourse-marker density discriminated.** A plain mean diluted the
one working feature.

The fix was to change Signal 2's *internal* combination to a **weighted sum** (markers 0.5,
burstiness 0.25, repetition 0.25), keeping the global 0.5/0.5 blend. The justification comes
straight from the signal's own documented blind spot — burstiness and TTR are length-confounded,
so they should carry less weight. After the fix, all three bands are reachable and scores match
intuition (table above).

**The multi-signal payoff, concretely:** the *formal-human* paragraph is an LLM **false
positive** — Signal 1 flags it at 0.80. Signal 2, correctly reading human-like burstiness and
vocabulary diversity (0.11), pulls the blend down to 0.46 = **uncertain**, preventing a confident
wrong "likely AI" label. This is exactly the false-positive scenario the design anticipated.

### If we deployed this for real

We'd treat distance-from-0.5 as a real probability only after **calibrating against labeled
data** (e.g. isotonic/Platt scaling), and we'd add a "**too short to score**" guard that forces
`uncertain` below a word-count floor instead of trusting a noisy stylometric reading.

## Transparency label (the three variants)

The label changes with the confidence score; `{score}` is `combined_score` as a percentage.
Exact text produced by [`make_label`](labels.py):

| Variant | Condition | Exact label text |
|---|---|---|
| **High-confidence AI** | `combined_score >= 0.70` | `This content is likely AI-generated ({score} confidence). Our detection system found strong indicators of machine authorship.` |
| **High-confidence human** | `combined_score <= 0.30` | `This content is likely human-written ({score} confidence that it is AI-generated, i.e. {100−score}% likely human). Our detection system found few indicators of machine authorship.` |
| **Uncertain** | `0.30 < combined_score < 0.70` | `We could not confidently determine whether this content is AI-generated or human-written ({score} confidence it is AI-generated). Treat this attribution with caution.` |

Real rendered examples from the test runs above:

> **AI (70%):** "This content is likely AI-generated (70% confidence). Our detection system found strong indicators of machine authorship."
>
> **Human (14%):** "This content is likely human-written (14% confidence that it is AI-generated, i.e. 86% likely human). Our detection system found few indicators of machine authorship."
>
> **Uncertain (46%):** "We could not confidently determine whether this content is AI-generated or human-written (46% confidence it is AI-generated). Treat this attribution with caution."

Design intent: every variant states a **plain-language verdict** *and* an explicit **number**,
and the uncertain variant explicitly tells the reader to treat the attribution with caution
rather than implying a false verdict.

## Appeals workflow

A creator who believes they were misclassified sends `content_id` + `creator_reasoning` to
`POST /appeal`. The system:

1. Looks up the `content_id` (404 if unknown).
2. Updates the stored submission's status to **`under_review`** and sets `appeal_filed: true`.
   The original label and score are **not** overwritten or recomputed (no automated
   re-classification — out of scope).
3. Appends an `appeal` entry to the audit log containing the creator's reasoning and a
   **snapshot of the original decision** (attribution, confidence, both signal scores), so a
   human reviewer sees the contested decision and the appeal side by side via `GET /appeals`.

```bash
curl.exe -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "c34a3dc7-...", "creator_reasoning": "I wrote this myself. I am a non-native English speaker and my style may appear more formal than typical."}'
# → {"appeal_id": "...", "content_id": "c34a3dc7-...", "status": "under_review", "message": "Appeal received. This content is now under review."}
```

This appeal scenario is deliberate: the formal-human paragraph that landed in "uncertain" is
exactly the kind of non-native/formal writing the LLM judge over-flags, which is the most likely
real-world appeal.

## Rate limiting

`/submit` is rate-limited with **Flask-Limiter** at **`10 per minute; 100 per day`**, keyed by
**client IP** (`get_remote_address`), using in-memory storage.

**Why these numbers:** a real writer submitting their own work rarely posts more than a few
pieces a minute, so 10/min leaves comfortable headroom for legitimate use while a flooding script
trips it immediately; 100/day caps sustained abuse from a single source. **Why key on IP and not
`creator_id`:** there is no authentication in scope, so `creator_id` is attacker-controlled and
trivially rotated — keying on it would not stop a flooding script. Client IP is the meaningful
throttle key. (In production with real auth, we'd key on authenticated identity *and* IP.)

**Evidence** — 12 rapid requests (limit is 10/min); first 10 succeed, the rest are throttled:

## Audit log

Every decision and appeal is written to SQLite and exposed as JSON at `GET /log`. Each
**submission** entry captures: timestamp, `content_id`, `creator_id`, attribution, combined
confidence, **both individual signal scores** (plus the LLM rationale and the stylometric feature
breakdown), and status. Each **appeal** entry captures: timestamp, `appeal_id`, `content_id`,
the creator's reasoning, status `under_review`, and a snapshot of the original decision — i.e.
the log records **whether an appeal has been filed**.

## Known limitations

These are tied to specific properties of the signals, not generic "needs more data":

1. **Formulaic poetry / intentionally simple writing (villanelles, children's verse).** Signal 2
   keys on low lexical diversity and uniform short sentences as "AI-like." A villanelle *repeats
   whole lines by definition* and a children's poem uses deliberately plain words, so Signal 2
   will push toward "AI" purely because of the literary form. The LLM judge may or may not
   rescue it — meaning this content is at real risk of a wrong or over-confident label.
2. **Non-native-English / formal human writing.** This is Signal 1's documented bias: the LLM
   judge over-flags formal or ESL prose as AI (we saw exactly this — the formal-human paragraph
   got 0.80 from the LLM). Signal 2 partially offsets it (pulling our example down to
   "uncertain"), but for a more marker-heavy formal text the blend could still cross into "likely
   AI" — a false positive against a real human writer. This is the primary appeal scenario the
   system is built to handle.
3. **Very short submissions (< ~40 words).** Signal 2's burstiness and TTR features are
   statistically meaningless on so little text, so confidence is unreliable. There is currently
   **no guard** that forces "uncertain" below a length floor — a known gap.
4. **Heavily human-edited AI (or AI-assisted human) text.** Neither signal is designed to detect
   a hybrid; the system tends to land in "uncertain," but it cannot distinguish "ambiguous
   because mixed-origin" from "ambiguous because we have no strong signal."

## Spec reflection

**One way the spec helped:** writing the detection-signals section in
[planning.md](planning.md) *before* coding — with each signal's exact output shape
(`{"ai_likelihood": float, "rationale": str}`) and the threshold table — meant the two signals
and the scoring function were defined as independent contracts. We could build and unit-test each
signal in isolation, then combine them, without rework. The spec's **false-positive trace**
(Section 10) also predicted the formal-human failure mode *before* we saw it in testing, so when
the LLM flagged that paragraph at 0.80 we already understood it and could point to Signal 2 doing
its job.

**One way the implementation diverged:** the spec specified Signal 2 as the **plain mean** of its
three sub-features. M4 testing showed this silently diverged from the *intent* — it diluted the
only feature that discriminated on short text, so even clearly-AI text scored "uncertain." We
changed the implementation to a **weighted sum** (markers 0.5 / burstiness 0.25 / repetition
0.25). The divergence was justified by the spec's own blind-spot analysis (burstiness and TTR are
length-confounded), and we updated planning.md to match. (A second, smaller divergence: rate
limiting is keyed by **IP** rather than the spec's draft `creator_id`, because without auth
`creator_id` can't prevent abuse.)

## AI usage

AI coding assistance was used throughout, directed by the per-section specs in planning.md. Two
specific instances where the AI's output was revised or overridden:

1. **LLM signal function (Signal 1).** Directed the AI to generate `get_llm_signal(text)` calling
   Groq in JSON mode with the prompt template from the spec. It produced a working shape but
   built the prompt with `str.format()`, which **crashed** (`KeyError: '"ai_likelihood"'`) the
   moment the prompt contained a literal JSON example with `{` `}` braces. **Override:** replaced
   `.format()` with a `__TEXT__` placeholder substitution so the literal JSON braces in the
   prompt are left untouched. Verified by calling the function directly on a known-human and
   known-AI text before wiring it into the route.

2. **Confidence scoring (combine + Signal 2).** Directed the AI to implement the stylometric
   signal and `combine_scores` per the spec, combining the sub-features with a mean. The output
   *looked* reasonable and matched the literal spec — but verification against the four
   calibration inputs showed it silently diverged from intent: clearly-AI text scored 0.639
   ("uncertain") and the "likely AI" band was unreachable. **Override:** after printing each
   signal separately to find the cause (length-confounded sub-features on short text), changed the
   stylometric combination to a weighted sum that gives discourse markers half the weight, and
   re-verified all three bands were reachable. This is the calibration story documented above.

A recurring pattern in both cases: the AI's code was accepted only **after independent
verification** (direct function calls, the four-input calibration set, the 12-request rate-limit
test), and the spec was updated whenever the implementation was deliberately changed.
